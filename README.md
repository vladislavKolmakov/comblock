# News Parser (тестовое задание)

Небольшой backend-сервис для хранения и обогащения новостей из `ria.ru`.
Сервис принимает список ссылок, запускает парсинг в фоне и сохраняет результат в PostgreSQL.

## Что внутри

- один FastAPI-сервис (без брокеров и очередей);
- PostgreSQL для хранения новостей и изображений;
- асинхронный доступ к БД через SQLAlchemy 2.0 + `AsyncSession`;
- миграции через Alembic;
- парсер на `httpx` + `BeautifulSoup4`;
- фоновая обработка через `BackgroundTasks`.

## Архитектура и ключевые решения

Проект разделен по слоям:

- `app/api` — HTTP-эндпоинты;
- `app/services` — бизнес-логика (обогащение новостей);
- `app/repositories` — работа с БД;
- `app/parsers` — парсинг внешнего источника;
- `app/database` — сессия;
- `app/models` — модели;
- `app/schemas` — Pydantic-схемы для API;
- `app/core` — конфигурация.

Модель данных:

- `News` — основная сущность новости;
- `NewsImage` — связанные изображения;
- связь `News 1 -> N NewsImage`.

Статусы обогащения:

- `SUCCESS` — данные успешно загружены и сохранены;
- `FAILED` — при ошибке сохраняется текст в `error_message`;
- `NULL` — запись еще не обогащалась.

## Почему именно так

- **Один сервис FastAPI** — для масштаба тестового задания этого достаточно, меньше инфраструктуры и проще запуск.
- **BackgroundTasks** — запрос на запуск обогащения возвращается сразу, а тяжелая работа уходит в фон.
- **Async SQLAlchemy + asyncpg** — нормальный вариант для I/O-нагруженного сценария (HTTP + БД).
- **Alembic вместо `create_all`** — схема БД фиксируется миграциями, её можно воспроизвести в любом окружении.
- **Repository + Service слой** — код проще поддерживать: отдельно API, отдельно бизнес-логика, отдельно запросы к БД.

## Устойчивость парсинга

Новостные сайты периодически меняют верстку, поэтому в парсере сделаны базовые меры устойчивости:

- используется несколько селекторов для ключевых полей (где это возможно);
- если поле не найдено, сохраняется `NULL` вместо падения обработки;
- ошибки по конкретному URL не валят весь процесс, а фиксируются в `enrichment_status=FAILED` и `error_message`.

Это не решает проблему изменений верстки на 100%, но дает предсказуемое поведение и понятную диагностику.

## Запуск

### Через Docker (основной способ)

1. Создать `.env`:

```bash
cp .env.example .env
```

2. Поднять сервисы:

```bash
docker compose up --build
```

После старта API доступен на `http://localhost:8000`.

### Локальные миграции Alembic

Создать миграцию:

```bash
alembic revision --autogenerate -m "create tables"
```

Применить миграции:

```bash
alembic upgrade head
```

## API документация

- Swagger UI: `http://localhost:8000/docs`
- OpenAPI JSON: `http://localhost:8000/openapi.json`

## Эндпоинты

- `GET /news` — список новостей (`limit`, `offset`);
- `GET /news/{id}` — одна новость;
- `POST /news/enrich` — запуск фонового обогащения.

## Примеры запросов

Запустить обогащение:

```bash
curl -X POST "http://localhost:8000/news/enrich" \
  -H "Content-Type: application/json" \
  -d '{
    "urls": [
      "https://ria.ru/20200101/example.html"
    ]
  }'
```

Получить список новостей:

```bash
curl "http://localhost:8000/news?limit=10&offset=0"
```

Получить новость по id:

```bash
curl "http://localhost:8000/news/1"
```
